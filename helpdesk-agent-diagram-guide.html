Scenario
Goal: public help desk agent that talks to users on Telegram / WhatsApp / iMessage, creates tickets, answers queries, and uses only self-hosted/free tools.
  • Primary channels: Telegram, WhatsApp, optional iMessage
  • Core behavior: ticket creation/updates, search, how-tos, triage
  • Constraints: no GPU, 4–8 GB RAM, 4 vCPU, fully self-hosted/free where possible
  • Non-negotiable: security, stability, sanity checks around public access

PII rules

  • Never auto-attach full Telegram metadata. Keep your own mapping if needed.
  • Strip URLs, scripts, and prompt-like text from ticket bodies.
  • Confirm before creating external tickets.
  • Rate-limit per user to avoid spam.

Use SQLite + llama.cpp embeddings, no cloud services.

Config

memory:
  backend: sqlite
  url: sqlite:////opt/hermes/data/memory.sqlite3
  embed_endpoint: http://127.0.0.1:8080/embedding
  embed_model: qwen2.5-7b-instruct-q4_k_m

Hygiene

  • Separate per-user short-term memory from global ticket memory.
  • Never merge them.
  • Scrub old threads via cron.
Best Free 64k-Context Model
Pick one path, based on your VM RAM. The rule I use: 64k context is doable if the working set fits with system overhead.
Hard rule
64k context needs more memory than 32k. The same model file uses noticeably more RAM at 64k than at 32k.
PathModelRAM neededStrength
4 GB / 4 vCPUQwen2.5-3B-Instruct Q4_K_M~2 GB model + ~1 GB system = fitsFastest, lowest OOM risk

llama.cpp server systemd unit (64k)

[Unit]
Description=llama.cpp server
After=network-online.target

[Service]
Type=simple
User=hermes
WorkingDirectory=/opt/llama.cpp
ExecStart=/opt/llama.cpp/bin/llama-server \
  --model /opt/llama.cpp/models/CHANGE_ME.gguf \
  --port 8080 \
  --host 127.0.0.1 \
  --ctx-size 65536 \
  --n-gpu-layers 0 \
  --parallel 1 \
  --embedding
Restart=always
RestartSec=3
Environment="LLAMA_LOG_LEVEL=info"

[Install]
WantedBy=multi-user.target

model:
  provider: llama_cpp
  name: REPLACE_MODEL_NAME
  base_url: http://127.0.0.1:8080
  context_size: 65536

Tip: start with 64k set in config, then lower it only if you hit startup OOM. Check with journalctl -u llama-cpp-server -f.

Cost model formula

def compute_monthly_savings(prompts, completions, searches, agents=1):
    llm = (prompts * 2.5 + completions * 10) / 1_000_000
    search = searches * 0.005
    seats = agents * 29
    hosted = llm + search + seats + 80
    return max(hosted, 0)

Security Model
This agent will be public-facing. Treat it like a shared service from day one.
Hard rules
Never run Hermes as root. Never expose internal APIs without auth and rate limiting. Never trust user input blindly.
  • Network: UFW deny all inbound except 22/80/443. Fail2ban on SSH.
  • App: required token/start flow for Telegram users or phone-based allowlist. Rate limit per user.
  • Tools: Hermes must only call its own tools and approved self-hosted services. No raw shell exec unless explicitly enabled.
  • Input safety: strip or escape injected commands, limit message size, normalize whitespace.
  • Logging: log every interaction to SQLite; never log secrets; rotate logs.
  • Updates: keep base OS/Hermes deps updated; audit installed packages.

First-Run Checklist
  • VM provisioned and SSH key-only access enabled
  • UFW enabled (22/80/443 only)
  • Domain + TLS via Let's Encrypt or Caddy
  • Hermes installed + systemd unit running + logs clean
  • Telegram bot configured and verified
  • osTicket / Zammad / email adapter configured
  • At least one ticket created and closed via agent
  • SearXNG up and responding on a subpath
  • Backup schedule for DB + configs defined and tested
  • Emergency stop: command to disable public chat intake exists
